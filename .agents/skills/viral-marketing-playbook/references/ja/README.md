# バイラルループ設計 — リファラル・招待・シェアの仕組み

プロダクトは偶然バズらない。バイラル性はプロダクトの中にエンジニアリングするもの——それがこのプレイブックの主題。

## バイラルループの解剖学

- 3 種類のループの仕組み：リファラルループ、招待ループ、コンテンツループ
- バイラルループはマーケティング施策ではなくプロダクト機能。ユーザージャーニーの内側に組み込む

## K ファクターの計算

- 自社のバイラル係数を算出し、そこから成長シナリオをモデリングする
- K は「1 コホートのユーザーが、獲得コスト以上のユーザーを連れてくるか」を教えてくれる。ループ機能に投資する前に必ず測る

## インセンティブ設計

- 紹介する側に何を渡し、紹介される側に何を渡すか——そして「効かないインセンティブ」はどれか
- 両面インセンティブは、双方がプロダクトの中で本当に価値を感じるものでなければ機能しない

## シェアのトリガー

- ユーザージャーニーの中で、シェアが「押し付け」ではなく「自然」に感じられる瞬間を見つける
- タイミングを間違えたシェア導線は摩擦になる。同じ導線でも、正しい瞬間なら成長エンジンになる

## ケーススタディ

- Dropbox・Notion・Loom はどうやってバイラルエンジンを作ったか
- 各エンジンの背後にある仕組みを学び、パターンを自社に適応させる。表面的な機能のコピーはしない

## B2B と 2C の使い分け

本プレイブックのデフォルトは開発者ツール／オープンソース／B2B SaaS。プロダクトが 2C（コンシューマー・教育・アプリ・ゲーム）の場合：

- コア指標を D1/D7/D30 リテンション・アクティベーション率・バイラル係数 K に置き換える（MRR/CAC/LTV ではなく）
- コールドスタートのチャネルを垂直コミュニティ・ショート動画・専門 KOL に置き換える（Product Hunt / Hacker News / LinkedIn ではなく）
- 国別チャネルデータを含む完全な 2C 適応ガイドは `gingiris-seo-geo/references/2c-adaptation.md` を参照
