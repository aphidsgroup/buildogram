# 开源发布 Pre-launch 完整清单 · 速查手册

> 提炼自 AFFiNE 0 → 60K star 实战。窗口：T-30 → T-0。
> Star 不来自"好 repo"，来自**把好 repo 按正确顺序放到对的开发者面前**——准备期决定顺序能不能执行。

## Pre-launch 总清单（T-30 → T-0）

- [ ] README **英文主**，首屏 <3 秒读懂
- [ ] Demo video ≤ 60s，带字幕
- [ ] License 有意识地选（MIT / Apache-2 / AGPL 各有陷阱，商业化路径不同）
- [ ] 3–5 个早鸟 maintainer / contributor 就位
- [ ] 社群频道上线（Discord 或 Telegram，英文为主）
- [ ] PR 文章写好（见下）
- [ ] 发布时序排好：第一周只做海外，第二周才做国内

## README 转化清单

- [ ] 结构：大 logo → 一句话介绍 → badges → 简短介绍（参考 Dify / Lobechat / AFFiNE）
- [ ] **不超过 8 屏**——开发者文档单独链接，不塞进 README
- [ ] 加**提醒点 star 的小动图**（类比 B 站"一键三连"——用户需要被提醒）
- [ ] **Backlinks 循环**：所有跳出去的链接（官网/博客/文档站）必须有明显的 GitHub 回链，流量不能有去无回
- [ ] ❌ 别堆太多 emoji / badge——降低可信度

## PR 文章：给传播者一把枪

- 发布前准备一篇完整阐述项目价值的权威文章（官网 Blog / Medium / dev.to）。
- 目的：任何想帮你传播的人可以直接引用，不需要自己想怎么介绍。
- AFFiNE 案例：一个德国用户没打招呼就独立发了 40+ 论坛——能发出去，是因为有参考文章。

## Star 地区分布（GitHub Trending 隐性门槛）

- 同一国家/地区 star 占比不超过 **20%**，否则难上 Trending。
- 健康分布参考：中国 19-21%、美国 19-21%，其余分散欧洲/俄罗斯/加拿大/印尼。
- 监控工具：**star-history.com**（涨幅趋势）· **OSS Insight**（国家分布）
- **发布时序是准备期就要锁死的决策**：第一周只做海外宣发，第二周才做国内。顺序反了会破坏分布比例，Trending 资格直接没了。
- AFFiNE 真实案例：开源第一周故意不发朋友圈、不进国内社群；第一周拿 6000 star 几乎全来自海外渠道。

## 渠道顺序原则

> **英文 > 多语言 > 纯中文。** 除非目标就是中文用户，否则把英文项目先发中文社群是浪费最好的一次机会。

## 节奏基准（陪跑项目参考）

- 至少提前 **2-3 个月**开始准备。
- 爆发期一周 **4000-6000 star**，三周可达 **10000 star**（AFFiNE：10K star / 43 天）。

## 准备期反模式

- ❌ 刷 star——GitHub 检测很快，后果是*从搜索里消失*
- ❌ README 没有英文版就发海外社区
- ❌ 发布日才建 Discord（社区冷场比没有更伤）
- ❌ 没有 PR 文章就指望别人替你介绍
