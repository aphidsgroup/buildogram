# 제품 × 개발 × 운영 협업 SOP

모든 것은 수익화를 위해. 통합 백로그부터 10일 스프린트 리듬, 릴리스 거부권까지 완전한 3자 협업 프레임워크.

## 왜 세 팀의 협업은 늘 어긋나는가

- **신기능 vs 사용자 피드백**: 개발팀은 신기능만 쫓고, 보고된 버그는 몇 달째 방치 — 통합된 우선순위 결정 메커니즘 부재
- **개발이 하고 싶은 것 vs 운영이 원하는 것**: 개발은 "운영이 기술을 모른다", 운영은 "개발이 사용자 말을 안 듣는다" — 공동 목표 부재
- **속도 vs 안정성**: 제품 형태가 안정되기 전에 성장을 서두르다 기술 부채만 쌓임 — 제품 단계 판단 기준 부재

해결의 4대 메커니즘: 통합 칸반, 스프린트마다 명확한 상업화 지표(예: 가입→결제 전환율 4.5% → 7%), 운영의 릴리스 거부권, 매 스프린트 전 3자 얼라인먼트.

## 2층 칸반 체계

- **총 백로그**: 모든 원시 이슈(사용자 피드백/경쟁사 분석/전략 요구)는 먼저 하나의 풀로
- **스프린트 보드**: 리뷰를 거쳐 일정이 확정된 항목만 하류로 유입
- **이슈 규율**: 운영이 등록하는 모든 이슈에는 재현 단계 필수 — 재현할 수 없으면 고칠 수 없음
- **자동 에스컬레이션**: 동일 이슈가 3회 이상 보고되면 `severe` 태그 부착, 가장 가까운 스프린트에서 반드시 해결
- **분류**: 버그는 버그 보드로 직행, 기능 요청은 제품 기획안이 나온 뒤에야 스프린트 진입

## 10일 스프린트 표준 리듬

- Day 1 개발 시작 → Day 6 테스트 가능 빌드 제출(운영은 반나절 내 테스트 완료) → Day 7 얼라인먼트 회의(20-30분) → Day 9 출시 자료 준비 → Day 10 릴리스
- 단계별 주기: PMF 검증기 1-2주, 성장기 3-4주, 안정기 6-8주 — 현대 웹 개발 리듬에서는 10일~2주로 압축 권장
- **레드라인**: 금요일이나 주말에는 절대 릴리스하지 않는다

## 회의 리듬

- **데일리 스탠드업**: 15분, 서서 진행. 어제/오늘/블로커 세 가지만. 운영은 전날 사용자 불만 핫스팟을 반드시 공유
- **스프린트 계획 회의**(30-60분): 지난 스프린트 회고 → 상업적 가치와 연결된 Top 3 목표 → 범위 확정
- **스프린트 검수 회의**(30분) + **월간 리뷰**(60-90분). 1시간 넘는 회의에는 반드시 딴짓하는 사람이 생긴다

## 운영의 거부권

UX나 결제 전환을 직접 해치는 P0 버그가 미해결이면 운영은 릴리스 연기를 제안할 수 있다. P0 = 결제 실패·핵심 기능 불능·데이터 오류. 레이아웃 깨짐은 P1, 문구 오타는 P2.

## 리소스 배분과 기술 부채

- 배분 기준: 신기능 50-60% / 버그 수정 20-30% / 기술 부채 20-30%
- 리팩토링은 별도 프로젝트로(1-3개월), 정상 릴리스와 병행 운영하며 완료 후 핵심 사용자 3-5명이 참여하는 완전 회귀 테스트 필수

## 스프린트 건강도 벤치마크

| 체크 항목 | 건강 기준 | 경고 신호 |
|----------|----------|----------|
| P0 버그 평균 수정 시간 | ≤2일 | >3일 |
| 기능 정시 출시율 | ≥80% | <60% |
| 피드백 평균 응답 시간 | ≤24시간 | >48시간 |
| 동일 버그 반복 보고 횟수 | ≤2회 | ≥3회 |
| 테스트 리포트 제출 | Day 6.5까지 100% | 반나절 이상 지연 |
